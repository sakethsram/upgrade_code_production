Here's the full descriptive breakdown — everything your manager needs to understand.

What the system does (big picture)
This is a network device upgrade automation tool. Before upgrading a router's OS, the tool runs a series of pre-checks — it SSHs into the device, runs a bunch of show commands, backs up config, checks storage, validates the image checksum, and disables a firewall filter. All of this happens in main.py inside a function called run_prechecks().

The pre-existing flow (before our changes)
When the tool connected to a device and ran show commands, here's what happened:
1. YAML files drove everything. Two YAML files live in the inputs/ folder — deviceDetails.yaml (has the device IP, credentials, vendor, model) and show_cmd_list.yaml (has the list of CLI commands to run, organized by vendor and model).
2. Commands were executed via execute_command() in utilities.py. This function took the command list from the YAML, looped through each command, sent it to the device via Netmiko (the SSH library), and got back raw CLI text output — the kind of thing you'd see if you were sitting at the terminal.
3. Outputs were stored in a flat, unstructured dict. The results went into global_config.pre_output which was just a plain dictionary — key was the command string, value was a nested dict with the device name and the raw output text. Something like:
{
  "show arp no-resolve": { "device": "juniper_mx204", "output": "...raw text..." },
  "show vrrp summary":   { "device": "juniper_mx204", "output": "...raw text..." }
}
4. Parsers existed but were disconnected. There was already a parsers/ folder with parser functions — for example parse_show_arp_no_resolve() in parsers/junos/junos_mx204.py. These parsers took raw CLI text and returned clean structured JSON (a list of ARP entries with MAC, IP, interface, flags). But — and this is the key problem — nobody was calling them during the pre-check workflow. They existed as standalone functions that were never wired into the main flow. The raw output was collected and just... sat there.
5. Data models existed too. There's a models/ folder with dataclasses like ShowArpNoResolve, ShowArpNoResolveEntry, ShowVrrpSummary etc. These defined the schema for what parsed output should look like. Again — existed, but weren't being used in the workflow.

What the problems were

After running pre-checks, you had raw CLI text sitting in a dict. If you wanted to compare pre and post-upgrade states (e.g., did the ARP table change?), you had no structured data to compare — just two blobs of text.
The parsers were written but never invoked automatically. Someone would have to manually call them, know which parser maps to which command, pass the right output in — it was all manual and error-prone.
There was no consistent structure — some parts of the code stored output one way, other parts another way. execute_command_test() (a test version of the function) stored things differently from execute_command().
Adding a new parser meant digging into multiple files to wire it up. There was no central place that said "this command uses this parser."


What we changed and added
Change 1 — Restructured the global storage
We changed GlobalConfig in utilities.py. Instead of pre_output being a flat dict, we introduced precheck_results as a list of structured entries. Every command now gets stored as:
python{
    "command":     "show arp no-resolve | no-more",   # the command that was run
    "output":      "...raw CLI text from device...",   # what the router returned
    "json_output": {}                                  # placeholder for parsed data
}
json_output starts empty. It gets filled in later by the parser. This means every command's raw output and parsed output travel together in one object — no cross-referencing between different dicts, no mismatch.
Same structure exists for postcheck_results for post-upgrade checks (wired up later).

Change 2 — Modified execute_command() in utilities.py
The function still does what it did — loops through commands from the YAML and sends them to the device. But now instead of dumping output into the old flat dict, it:

Resets global_config.precheck_results to an empty list at the start of each run (clean slate)
For each command, builds one of those structured entries
If the command succeeds, output gets the CLI text
If the command fails, output stays empty and json_output gets an error message
Appends the entry to precheck_results
Returns the full list at the end

So after execute_command() runs, you have a clean list of every command with its raw output, ready to be parsed.

Change 3 — New file: parsers/parser_wrapper.py
This is the biggest addition. We created a new file that acts as the bridge between command execution and parsing. It has two things:
The registry — a dictionary called PARSER_REGISTRY that maps a tuple of (vendor_model, command_string) to the actual parser function:
pythonPARSER_REGISTRY = {
    ("juniper_mx204", "show arp no-resolve | no-more"): parse_show_arp_no_resolve,
    ("juniper_mx204", "show vrrp summary | no-more"):   parse_show_vrrp_summary,
}
```

This is the central lookup table. If you add a new parser tomorrow, you add one line here. Nothing else changes.

**The wrapper function `run_parsers(check_type)`** — takes either `"pre"` or `"post"` as input, picks the right results list (`precheck_results` or `postcheck_results`), and then:

- Loops through every entry in that list
- Skips entries where output is empty (command failed, nothing to parse)
- Looks up `(vendor_model, command)` in the registry
- If a parser is found — calls it, takes the returned JSON, stores it in `entry["json_output"]`
- If no parser is found — logs a warning and moves on (non-fatal, workflow doesn't die)
- If the parser itself throws an exception — catches it, stores the error in `json_output`, logs it, and continues

So after `run_parsers("pre")` runs, every entry in `precheck_results` that has a registered parser now has structured JSON sitting in `json_output`.

---

### Change 4 — Wired it all together in `run_prechecks()` in `main.py`

In the pre-check orchestration function, Step 3 now does this in sequence:

1. Load the YAML command list
2. Call `execute_command()` → fills `precheck_results` with raw outputs
3. Immediately call `run_parsers("pre")` → fills in `json_output` for every parseable command
4. Log a debug line per command showing what was captured and what keys came back from parsing
5. Continue to backup, checksum, storage, filter steps as before

If parsing partially fails (one parser errors out), the workflow **does not abort** — it logs a warning and continues. Parsing failure is non-fatal because the raw output is still there. If command execution itself fails, that's fatal and triggers `_abort()`.
```
